# Техническое задание
## Интернет-магазин инвестиционного золота

**Версия:** 1.0  
**Дата:** 2026  
**Статус:** Черновик

---

## 1. Общая информация

### 1.1 Назначение документа
Настоящее ТЗ описывает требования к разработке веб-сайта для продажи инвестиционного золота: слитков, монет и пластин. Документ является основой для проектирования, разработки, тестирования и приёмки системы.

### 1.2 Цели проекта
- Обеспечить онлайн-продажу физического инвестиционного золота
- Отображать актуальные биржевые котировки в режиме реального времени
- Автоматически пересчитывать цены товаров на основе котировок
- Обеспечить надёжный и безопасный процесс оплаты
- Предоставить администратору удобный инструмент управления

### 1.3 Целевая аудитория
- Частные инвесторы (25–65 лет), покупающие золото как защитный актив
- Корпоративные покупатели (малый и средний бизнес)
- Коллекционеры монет

## 3. Функциональные требования

### 3.1 Модуль каталога

**3.1.1 Список товаров**
- Постраничный вывод (пагинация или infinite scroll)
- Фильтрация по: типу, весу, производителю, цене
- Сортировка по: цене (возр./убыв.), популярности, новизне
- Режимы отображения: сетка / список

**3.1.2 Карточка товара**
- Главное фото + галерея (до 8 изображений)
- Название, описание, характеристики
- Цена покупки и цена выкупа (в RSD, EUR, USD)
- Индикатор наличия на складе
- Кнопки: "В корзину", "В wishlist"
- Хлебные крошки

**3.1.3 Поиск**
- Полнотекстовый поиск по названию и описанию
- Автодополнение при вводе (debounce 300мс)
- История поиска (localStorage)

---

### 3.2 Модуль котировок

**3.2.1 Отображение**
- Тикер в шапке: XAU/RSD, XAU/EUR, XAU/USD
- Индикатор изменения за день (%, абсолютное значение)
- Стрелка вверх/вниз с цветовой индикацией

**3.2.2 Обновление**
- Polling каждые 60 секунд через API
- WebSocket для real-time если API поддерживает
- Fallback на последнее известное значение при недоступности

**3.2.3 Ценообразование**
- Цена товара = (спотовая цена × вес в тр. унциях) × наценка администратора (%)
- Наценка настраивается в панели администратора для каждой категории
- Пересчёт при каждом обновлении котировки

**3.2.4 График**
- Исторические данные: 1Д, 1Н, 1М, 6М, 1Г, 5Л
- Интерактивный (tooltip при наведении)
- Библиотека: Recharts или Lightweight Charts (TradingView)

---

### 3.3 Модуль корзины

**3.3.1 Функциональность**
- Добавление/удаление товаров
- Изменение количества
- Сохранение для гостей (localStorage, 24ч)
- Синхронизация с БД после авторизации
- Уведомление об изменении цены

**3.3.2 Расчёты**
- Промежуточная сумма по каждой позиции
- Итоговая сумма
- Стоимость доставки (рассчитывается на следующем шаге)

---

### 3.4 Модуль оформления заказа

**3.4.1 Шаги (Wizard)**
1. Корзина → 2. Контакты → 3. Доставка → 4. Оплата → 5. Подтверждение

**3.4.2 Способы доставки**
- Курьерская (по городу)
- Почтовая (по стране)
- Самовывоз из офиса

**3.4.3 Способы оплаты**
- Банковская карта (Stripe / NestPay)
- Банковский перевод (инструкции на email)
- Наличные (только при самовывозе)

**3.4.4 Статусы заказа**
`Новый` → `Оплачен` → `Подготовлен` → `Отправлен` → `Доставлен` → `Завершён`
`Отменён` (из любого статуса до отправки)

---

### 3.5 Модуль аккаунта

**3.5.1 Регистрация / Вход**
- Email + пароль
- OAuth: Google, Facebook
- Подтверждение email
- Восстановление пароля

**3.5.2 Личный кабинет**
- История заказов с деталями
- Управление адресами
- Список желаний
- Настройки профиля и пароля
- Подписка на уведомления

---

### 3.6 Контентный модуль

**3.6.1 Страницы**
- Главная (hero, преимущества, популярные товары, котировки, блог)
- О компании
- Как купить
- Обмен / Выкуп
- FAQ
- Контакты
- SEO-страницы по городам (Белград, Нови-Сад и др.)

**3.6.2 Блог**
- Список статей с фильтром по категориям
- Страница статьи с SEO-мета
- Похожие статьи
- Поиск по блогу

---

### 3.7 Панель администратора

**3.7.1 Управление товарами**
- CRUD: создать, редактировать, удалить товар
- Загрузка изображений (до 10 на товар)

**3.7.2 Управление заказами**
- Список заказов с поиском и фильтрами

- Печать накладной/чека

**3.7.3 Контент**
- Редактирование статических страниц
- CRUD для статей блога
- Управление FAQ
- Загрузка изображений для баннеров

**3.7.4 Аналитика**
- Дашборд: выручка, количество заказов, топ товары
- Фильтрация по периоду
- Экспорт в CSV

## 4. Нефункциональные требования

### 4.1 Производительность
| Метрика | Требование |
|---|---|
| LCP (Largest Contentful Paint) | < 2.5с |
| FID (First Input Delay) | < 100мс |
| CLS (Cumulative Layout Shift) | < 0.1 |
| Lighthouse Score | > 85 по всем категориям |
| API Response Time (p95) | < 500мс |

### 4.2 Безопасность
- HTTPS обязателен, HTTP → редирект на HTTPS
- Все формы защищены от CSRF (токены)
- Входные данные валидируются на клиенте и сервере (Zod)
- SQL-инъекции исключены через Prisma ORM
- Rate limiting на API endpoints
- Платёжные данные не хранятся на сервере (PCI DSS)
- Пароли хранятся в bcrypt (cost factor ≥ 12)
- Content Security Policy headers

### 4.3 Доступность
- Соответствие WCAG 2.1 уровня AA
- Навигация с клавиатуры
- ARIA-метки на интерактивных элементах
- Достаточный цветовой контраст (≥ 4.5:1)

### 4.4 SEO
- SSR для всех публичных страниц
- Статическая генерация для страниц товаров и блога
- Мета-теги: title, description, Open Graph, Twitter Card
- Structured Data (JSON-LD): Product, Organization, BreadcrumbList
- Sitemap XML, robots.txt
- Канонические URL

### 4.5 Совместимость
- Браузеры: Chrome 90+, Firefox 88+, Safari 14+, Edge 90+
- Устройства: Desktop (1920px), Tablet (768px), Mobile (320px–428px)
- ОС: Windows, macOS, iOS 14+, Android 10+

### 4.6 Надёжность
- Uptime: ≥ 99.5%
- Backup БД: ежедневно, хранение 30 дней
- Мониторинг: Sentry (ошибки), Vercel Analytics (производительность)
- Логирование всех транзакций

## 10. Приёмка и тестирование

### 10.1 Типы тестирования
- **Unit tests**: бизнес-логика (ценообразование, расчёты корзины) — Jest
- **Integration tests**: API endpoints — Jest + Supertest
- **E2E tests**: критичные пользовательские сценарии — Playwright
- **Performance**: Lighthouse CI в пайплайне
- **Security**: OWASP ZAP сканирование перед релизом

### 10.2 Критичные E2E сценарии
1. Гость → каталог → товар → корзина → оформление → оплата → email
2. Регистрация → вход → история заказов
3. Администратор → добавить товар → проверить в каталоге
4. Проверка обновления котировок на странице

### 10.3 Критерии приёмки
- Все Must Have user stories реализованы и соответствуют AC
- Core Web Vitals в зелёной зоне
- Прошли все E2E тесты на staging
- Security-сканирование без критичных уязвимостей
- Клиент подтвердил финальную версию на staging
